### 1 分钟回答版本

这个项目叫 `myagent`，是我自己做的一个本地优先代码仓库智能助手。  
我做它的出发点是：普通 LLM 虽然能回答代码问题，但很多时候只是文本层面的猜测，并不能真正理解本地项目、调用工具、读取代码、执行验证，所以我想做一个带有完整运行时内核的 Agent 系统。

我觉得这个项目最有价值的点，不是调用了大模型，而是我把它做成了一个可运行、可扩展、可调试、可验证的 Agent 内核。现在它已经能作为本地代码理解助手使用，后续我会继续把它往 Coding Agent 方向演化，比如增加代码编辑、测试验证和 Git 感知能力。

## Q2：这个项目的核心技术亮点是什么？

### 简短回答版本

我觉得这个项目的核心亮点主要有四个：

1. 我不是只做了一个聊天壳，而是自己实现了 Agent Runtime，有多轮 tool calling 和执行控制。
2. 我把模型能力和本地工具结合起来了，让它能真实检查代码仓库，而不是只靠猜。
3. 我做了 Memory、Skill、MCP 这些扩展层，让它具备持续对话、任务记忆和外部工具接入能力。
4. 我比较重视工程可用性，所以补了日志、错误处理、兼容适配和测试，而不是只停留在 demo。

### 展开回答版本

如果从工程角度讲，我觉得这个项目的亮点不在于“接了大模型”，而在于我把它做成了一个相对完整的本地智能体内核。

第一，Agent Runtime 是自己搭的。  
它不是单轮请求，而是支持模型先分析任务，再决定要不要调工具，然后把工具结果回填给模型继续推理，形成一个多轮闭环。

第二，工具系统是结构化的。  
我没有把工具调用写死在 prompt 里，而是抽象成统一的 ToolSpec / ToolResult 协议，让内置工具、本地工具和 MCP 工具都能接到同一套机制里。

第三，我专门做了上下文管理。  
除了 session 持久化，我还做了 memory v2，包括 summary、facts、task memory 和最小长期记忆，目的是让它在连续开发任务里不是每轮都从零开始。

第四，我比较重视可观测性和兼容性。  
因为做这种 Agent 最大的问题之一不是“能不能调通”，而是“出问题时能不能定位”。所以我做了 JSONL 日志、provider 原始响应 dump、OpenAI-compatible 双模式适配，以及比较完整的测试覆盖。

所以如果让我总结一句话，我会说这个项目的核心亮点是：**它已经不是一个 prompt demo，而是一个具备运行时、工具层、记忆层和扩展层的本地 Agent 系统。**

## Q3：你这个 Agent Runtime 是怎么实现的？

### 简短回答版本

我的 Agent Runtime 核心是一个多轮循环：

1. 先加载 session、system prompt、memory、skill prompt
2. 把用户输入拼到上下文里
3. 调 provider 拿模型响应
4. 如果模型返回 tool calls，就执行工具并把结果写回消息流
5. 如果没有 tool calls，就结束当前轮并保存结果

同时我还加了重复工具调用保护、单轮失败预算和连续失败保护，避免它卡死在无效循环里。

### 展开回答版本

这个 Runtime 的设计重点是把“聊天”升级成“任务执行循环”。

我会先把当前 session 的历史消息加载出来，然后把 system prompt、skill prompt 和 memory prompt 组合成最上层上下文，再接上用户当前输入。

之后进入一个最多 N 轮的 tool loop：

- 每一轮先调用 provider
- provider 返回普通文本时，说明模型已经得出最终结论，就直接落库并返回
- provider 返回 tool calls 时，就进入工具执行阶段
- 工具执行结果会被封装成统一结构，再作为 `tool` 消息回填给模型
- 模型再基于这些真实结果继续推理

这里面我专门做了几个保护机制：

- 相同工具相同参数重复调用保护
- 单轮工具失败预算
- 连续工具失败上限
- 每轮执行日志记录

这样做的目的，是让它不只是“能调工具”，而是能以比较工程化的方式调工具。

## Q4：工具系统是怎么设计的？

### 简短回答版本

工具系统我做成了统一抽象，核心是 `ToolSpec` 和 `ToolResult`。

- `ToolSpec` 负责描述工具名、参数 schema、说明和 handler
- `ToolResult` 统一封装工具执行结果，包括成功失败、文本输出、错误类型和元数据

这样内置工具、本地工具和 MCP 工具都能走一套统一执行链路。

### 展开回答版本

我在工具层的目标，是把“模型可调用的能力”做成标准化接口，而不是每个工具各写一套逻辑。

所以我抽象了两个核心对象：

1. `ToolSpec`
- 工具名
- 描述
- 参数 schema
- 实际执行 handler

2. `ToolResult`
- `ok / error`
- 文本内容
- `error_type`
- metadata
- 可选 structured payload

这样做有几个好处：

- provider 层只关心工具定义，不关心具体实现
- agent loop 只关心工具结果，不关心具体工具内部细节
- MCP 工具也可以被包一层后直接接入

目前我的内置工具包括：

- `list_dir`
- `repo_search`
- `read_file`
- `run_command`

而且我在工具层加了工作区路径限制、命令 allow/block 策略、输出截断、错误分类这些安全控制。

## Q5：Memory 是怎么实现的？

### 简短回答版本

Memory 这一层我做了三部分：

1. session transcript 持久化
2. memory snapshot，包括 summary、facts、task memory
3. 最小长期记忆，用来跨 session 复用稳定事实和任务结论

它的作用是让 agent 连续工作时，能记住上下文，而不是每一轮都从零开始。

### 展开回答版本

Memory 不是单纯把历史消息全塞回模型，因为那样上下文会越来越重，而且效果不稳定。

所以我做了 memory v2，大致分成三层：

第一层是 session transcript。  
完整消息历史会保存在本地，作为最原始记录。

第二层是 memory snapshot。  
这里面会维护：

- summary
- facts
- task memory

其中 task memory 会记录当前任务标题、目标、完成步骤、待办和 blocker。

第三层是最小长期记忆。  
我把相对稳定的事实和任务结论沉淀下来，让不同 session 之间也能复用。

真正用的时候，不是把所有 memory 全塞回 prompt，而是按当前 query 做相关性筛选，再动态注入相关 summary / facts / task notes。

这个设计的重点是：**控制上下文成本，同时保证连续任务的稳定性。**

## Q6：MCP 是怎么接入的？

### 简短回答版本

我接的是 `stdio` 模式的 MCP。  
核心做法是自己实现一个最小 MCP client，支持：

- `initialize`
- `tools/list`
- `tools/call`

然后把 MCP 暴露出来的工具统一包装成现有的 `ToolSpec`，这样模型就可以像调内置工具一样去调 MCP 工具。

### 展开回答版本

我接 MCP 的目的，不是为了追热点，而是为了让 Agent 的能力扩展更标准化。

实现上我做了一个最小 `stdio` MCP client：

- 启动子进程
- 用 JSON-RPC 风格消息通信
- 先走 `initialize`
- 再走 `tools/list`
- 最后支持 `tools/call`

拿到 MCP 工具定义后，我会把它们包装成统一的 `ToolSpec`，命名形式类似：

`mcp__<server_name>__<tool_name>`

这样在 Agent 看来，MCP 工具和内置工具没有本质区别，依旧走统一的工具执行和日志链路。

我还支持多 MCP server，并做了命名空间隔离，避免工具名冲突。

## Q7：你是怎么做可观测性和问题排查的？

### 简短回答版本

我做了两层可观测性：

1. session 级 JSONL 运行日志
2. provider 原始响应 dump

这样当第三方接口兼容性有问题，或者模型行为异常时，我可以快速区分到底是 prompt、tool、provider 还是上游返回格式的问题。

### 展开回答版本

这类项目最容易踩坑的地方，不是写功能，而是“出了问题不知道哪层坏了”。

所以我专门加了 observability：

- 每个 session 记录 JSONL 事件日志
- 记录 `turn_start`
- 记录 provider request / response
- 记录 tool result
- 记录 latency、retry、error metadata

另外 provider 层我还会把最近一次原始响应 dump 下来，方便看第三方 OpenAI-compatible 服务到底返回了什么。

这个能力在调第三方 base_url 的时候非常有用，因为有些服务表面兼容 OpenAI，实际上会返回 HTML、字符串或者非标准 JSON，如果没有 raw dump 很难排查。

## Q9：你为什么不直接用 LangChain / LangGraph？

### 简短回答版本

我不是反对这些框架，而是我这个项目当前阶段更想先把 Agent 的底层运行机制吃透，所以我优先选择自己实现一版最小内核。

这样做的好处是：

- 我对消息流、tool loop、memory、provider 适配都能完全掌控
- 出问题时更容易定位
- 后面要不要接 LangGraph，是我有选择权，而不是一开始就被框架绑定

### 展开回答版本

这个问题我其实想得比较清楚。  
像 LangChain / LangGraph 这类框架确实能提升开发速度，但它们也会提前替你做很多抽象，比如 agent state、tool orchestration、memory integration、execution graph 等。

而我做这个项目的目标，不只是“快速做出一个 demo”，而是想真正理解一个 Agent 系统底层是怎么工作的，所以我刻意选择自己把这些基础层搭起来。

比如在我这个项目里，下面这些都是我自己实现和控制的：

- runtime loop
- provider abstraction
- tool schema 与结果协议
- memory 注入策略
- skill prompt 注入
- MCP 工具接入
- 原始响应调试和日志追踪

这样做有两个价值：

第一，我对系统的控制力更强。  
比如第三方 OpenAI-compatible provider 出现兼容问题时，我可以直接看原始 payload、改 provider parser，而不是被框架黑盒限制。

第二，这对我理解 Agent 设计本身很重要。  
我认为如果一开始全交给框架，很容易“会用，但不会解释”，而我更希望这个项目在面试里是一个我能讲清楚设计和取舍的系统。

当然，我不是排斥 LangGraph。  
如果未来这个项目真的发展到复杂工作流、多节点状态机、多 Agent 协作，我会考虑引入图编排框架。但在当前阶段，我更看重对内核机制的掌控。

## Q10：这个项目里你觉得最难的技术问题是什么？

### 简短回答版本

我觉得最难的不是接模型，而是**让系统在真实环境里稳定工作**。  
具体来说最难的是三件事：

1. 第三方 OpenAI-compatible 接口兼容问题
2. 多轮 tool calling 的稳定性控制
3. 在有限上下文里做 memory 注入而不把 prompt 搞乱

### 展开回答版本

如果只是做一个最简单的 LLM demo，其实不难。  
真正难的是把它做成一个在真实开发环境里能稳定运行的 Agent。

我觉得我碰到最有挑战的点主要有三个：

第一，provider 兼容性问题。  
表面上很多服务都说自己兼容 OpenAI，但实际返回格式并不完全一致，有的支持 `responses`，有的只支持 `chat completions`，有的甚至配置错 URL 时直接返回 HTML 页面。  
这个问题逼着我把 provider 层做成双模式适配，并增加 raw response dump 和更稳的错误分类。

第二，tool loop 的稳定性。  
模型一旦能调工具，就会出现重复调用、无意义调用、连续失败循环这些问题。  
所以我加了重复工具调用保护、单轮失败预算、连续失败上限，本质上是在给模型行为加工程化护栏。

第三，memory 的上下文控制。  
不是 memory 越多越好，如果把所有历史都塞回去，prompt 会越来越重，效果也会变差。  
所以我做了 summary、facts、task memory 和长短期分层，并按 query 做相关性筛选，这里面其实是一个“上下文成本和效果”的平衡问题。

所以如果面试官问我最难的点，我会回答：  
**难点不在单个功能，而在于让模型、工具、上下文和真实系统边界一起稳定工作。**

## Q11：你是怎么保证这个 Agent 不会乱执行命令的？

### 简短回答版本

我没有把 `run_command` 做成完全开放，而是加了比较严格的约束：

- 只允许在 workspace 内执行
- 有 allowlist / blocklist
- 禁止危险操作符
- 有超时控制
- 输出长度截断

所以它不是一个全权限 shell，而是一个受控执行工具。

### 展开回答版本

这个问题我在设计时就比较谨慎，因为只要一开放命令执行，风险就会迅速放大。

所以我做了几层限制：

第一层是工作区边界。  
所有文件类工具都只能在 workspace 里操作，不能逃逸到外部路径。

第二层是命令策略。  
我没有允许任意命令执行，而是加了 allowlist / blocklist，比如：

- 禁止删除、重命名、重置这类高风险命令
- 禁止管道、重定向、链式操作这类复杂命令结构
- 只允许一小部分明确的命令前缀

第三层是执行控制。  
命令有 timeout，也有输出截断，避免卡死或把大量无关输出塞回模型。

第四层是结果结构化。  
命令执行结果会被包装成 `ToolResult`，包含状态、内容、错误类型和 metadata，便于 agent 判断下一步。

从设计上说，我把它当作“受控观察工具”，而不是“让模型随意拿 shell 权限”。

## Q12：如果让你把这个项目真正做成 Coding Agent，你下一步会怎么推进？

### 简短回答版本

我会优先补三条链路：

1. 文件编辑能力
2. 测试/验证能力
3. Git 感知能力

因为这三件事补齐之后，它才真正从“代码理解助手”变成“能完成编码闭环的 Agent”。

### 展开回答版本

现在这个项目更像一个代码仓库智能助手，它已经能：

- 看代码
- 搜代码
- 读文件
- 调工具
- 记上下文

但距离 Coding Agent 还差“动手改代码并验证”的最后一段。

我下一步会严格按这个顺序做：

第一，文件编辑工具。  
比如 `apply_patch`、`write_file`、`replace_in_file`，让它具备真正改代码的能力。

第二，验证工具。  
补 `run_tests`、`run_lint`、`run_typecheck` 这种语义化验证工具，而不是只靠通用 shell。

第三，Git 感知。  
增加 `git status`、`git diff`、`git log` 这种只读 Git 工具，让它知道自己改了什么，也避免覆盖用户已有修改。

第四，面向编码任务的 loop。  
把执行模式从一般对话 loop 升级成：

- inspect
- edit
- verify
- summarize

这种更偏工程任务的流程。

所以如果要继续往前推，我会把重点放在**代码修改闭环**，而不是继续加更多聊天层能力。

## Q13：这个项目里有哪些工程取舍？

### 简短回答版本

我做过几个比较明确的取舍：

- 先做本地优先，不先做 UI
- 先做内核，不先做平台化
- 先做最小长期记忆，不先做复杂向量检索
- 先做可解释和可调试，不追求特别重的框架封装

### 展开回答版本

我觉得这个项目里比较重要的一点是，我没有一开始就把它做成“大而全”，而是做了很多偏工程化的取舍。

比如：

第一，先本地优先，后平台化。  
我先把 CLI、HTTP API、工具调用、memory、MCP 跑通，而不是一上来就做 Web UI、账号体系或多 channel。

第二，先内核，后生态。  
我先做 runtime、provider、tool、memory 这些真正决定系统行为的部分，而不是先去套成熟框架或堆功能。

第三，先最小长期记忆，后复杂检索。  
我先做稳定事实和任务结论沉淀，再考虑更复杂的 memory consolidation 或向量检索。

第四，先可调试，后炫技。  
我很早就加了日志、raw dump、错误分类和测试，因为这种项目如果不能排查问题，后面根本走不远。

这些取舍本质上是为了让项目先具备“能持续演进”的工程基础。

## Q14：如果让你重新做一次，你会怎么优化这个项目？

### 简短回答版本

如果重新做一次，我会更早把两件事抽象清楚：

1. coding task state machine
2. 文件编辑/验证工具链

因为这两块决定了它能不能从助手升级成真正的 Coding Agent。

### 展开回答版本

如果让我回头看，我觉得现在这个项目的很多基础层是合理的，但如果重来一次，我会更早把“面向编码任务”的结构建出来。

比如：

第一，我会更早引入 coding task state machine。  
现在的 loop 已经能支撑工具调用，但编码任务天然更适合 `inspect -> edit -> verify -> summarize` 这种更明确的阶段结构。

第二，我会更早做文件编辑和验证工具。  
因为只要没有编辑能力和测试闭环，项目就更偏“代码理解助手”，而不是 Coding Agent。

第三，我会更早区分“聊天类 memory”和“工程任务 memory”。  
这两者的需求其实不一样，后者更强调 task progress、失败历史和验证状态。

第四，我会更早统一文档和编码规范。  
现在仓库里其实已经有比较多状态文档了，如果更早统一结构，后续多人协作和多机开发会更顺。

所以如果重新做一次，我会更早围绕“编码闭环”做系统设计。

## Q16：你这个项目是否涉及 ReAct Agent 设计模式？

### 简短回答版本

涉及，而且已经实现了 ReAct 的核心执行思想，但还不是论文式那种显式打印 `Thought / Action / Observation` 的完整 ReAct Agent。

更准确地说，我这个项目现在是一个 **具备 ReAct 内核特征的工具型 Agent Runtime**。

### 展开回答版本

如果从执行机制看，我这个项目本质上已经很接近 ReAct。

ReAct 的核心是：

- Thought
- Action
- Observation

而我这个项目里已经有比较清晰的对应关系。

第一步是 Thought。  
模型会先拿到 system prompt、memory、skill prompt、历史消息和当前用户输入，然后判断下一步到底是直接回答，还是调用工具。  
这部分虽然没有显式输出成 `Thought: ...` 文本，但本质上是存在的。

第二步是 Action。  
如果模型决定调用工具，就会产生结构化 tool call，比如去调用 `list_dir`、`repo_search`、`read_file`、`run_command`，这就是 ReAct 里的 Action。

第三步是 Observation。  
工具执行以后会返回结构化结果，再被塞回消息流，模型基于这些真实观察结果继续推理，这就是 Observation。

所以从 Runtime 机制上说，它已经实现了：

**推理 -> 工具调用 -> 结果观察 -> 再推理**

这个闭环。

### 和经典 ReAct 的区别

我这个项目和论文式 ReAct 最大的区别是：

- `Action` 和 `Observation` 是明确结构化的
- `Thought` 更多是隐式推理，而不是显式文本链

也就是说，我没有强制模型每一轮都输出：

- `Thought: ...`
- `Action: ...`
- `Observation: ...`

而是把它做成了更工程化的 Runtime：

- 模型决定是否调用工具
- 工具调用走统一协议
- 工具结果走统一回填
- 日志里保留执行轨迹

所以如果面试官问我“你是不是用了 ReAct”，我会回答：

**是的，这个项目已经采用了 ReAct 的核心思路，但实现方式更偏结构化和工程化，而不是论文里那种显式文本链。**

### 如果继续往 ReAct 扩展，我会怎么做

如果继续强化 ReAct 特征，我下一步会补这些能力：

1. 显式记录每轮 reasoning note  
把当前这一步为什么要调某个工具，作为结构化 thought summary 保存下来。

2. 显式 observation summary  
不仅回填工具原始结果，还自动生成一层 observation 级摘要。

3. 更清晰的终止条件  
除了“没有 tool call 就结束”，还可以有任务完成判定、信息充分判定和失败退出判定。

4. Reflection / ReAct 扩展  
当出现连续失败时，不只是继续重试，而是要求模型先反思前一步哪里错了，再继续规划。

### 收束表达

所以我会把这个项目定义成：

**一个采用 ReAct 风格多轮工具调用机制的本地 Agent Runtime。**

它已经具备 ReAct 的核心循环，只是当前更强调结构化执行和工程可控性，而不是显式展示推理链文本。
